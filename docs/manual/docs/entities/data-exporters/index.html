Data Exporters

Otoroshi is not a monitoring tool, but as a reverse proxy sitting in front of all your services it naturally observes everything: every HTTP request, every admin action, every backend health check, every security alert. Data exporters are the bridge between this rich stream of telemetry and your observability stack. They push events out of Otoroshi and into the external systems where you actually analyze, visualize, and alert on them -- Elasticsearch, Kafka, Datadog, Pulsar, and many others.

Without data exporters, events live only in memory and are lost when Otoroshi restarts. In production, you almost always want at least one exporter configured so that you have persistent visibility into what is flowing through the gateway.

Otoroshi supports a wide range of export destinations out of the box:

  • Search and analytics: Elasticsearch, Splunk, Datadog, New Relic
  • Streaming platforms: Kafka, Pulsar, JMS
  • Observability protocols: OTLP Metrics, OTLP Logs, custom Prometheus metrics
  • Storage: PostgreSQL, file, S3
  • Network: webhook (HTTP), TCP, UDP, Syslog
  • Traffic replay: GoReplay (file or S3)
  • Notifications: email via Mailgun, Mailjet, Sendgrid, or generic HTTP mailer
  • Programmable: WASM plugins, workflow processing, custom exporter plugins
  • Development: console (stdout)

Each exporter runs its own independent pipeline with configurable filtering (which events to keep), projection (which fields to include), and batching (how many events to group before sending). You can run as many exporters in parallel as you need, each with a different destination and different rules.

To try them out, you can follow this tutorial.

Why use data exporters?​

Otoroshi generates a continuous stream of events as it processes traffic: every proxied request, every admin action, every alert, every health check result. By default these events live only in memory and are lost when the instance restarts. Data exporters let you push those events to external systems so you can:

  • Monitor traffic in real time (dashboards, alerting)
  • Audit admin activity (who changed what, and when)
  • Debug issues by replaying request/response flows
  • Meet compliance requirements with long-term event archival
  • Feed custom analytics pipelines (data lakes, ML, billing)

You can create as many exporters as needed. Each exporter runs independently with its own destination, filtering rules, and batching settings. For example, you might send all traffic events to Elasticsearch for dashboards, alerts to a Slack webhook, and audit events to S3 for long-term storage, all at the same time.

Event types​

Otoroshi produces several categories of events:

Event typeDescription
GatewayEventEmitted for every proxied HTTP request. Contains full request/response metadata: status code, latency, headers, upstream target, etc. This is the highest-volume event type.
AlertEventTriggered when something notable happens: a circuit breaker opens, a quota is exceeded, an API key is about to rotate, a certificate is expiring, etc.
AuditEventTracks admin operations: back-office logins, API key creation, route changes, and any admin API call.
HealthCheckEventResult of periodic backend health checks (up/down, response time).
TrafficCaptureEventFull request and response body capture (only when the capture flag is enabled on a route). Used with GoReplay exporters.
TcpEventEvents from the TCP proxy (non-HTTP traffic).
RequestBodyEvent / ResponseBodyEventRequest or response body content, captured by specific plugins.
Other events from plugins or extensionsfor anything useful

Each event is enriched before export with additional context: geolocation data derived from client IP, parsed user-agent details, Otoroshi instance metadata (name, zone, region, datacenter), and cluster information.

How the pipeline works​

Each data exporter runs an independent streaming pipeline built on top of Akka Streams:

Events → Queue → JSON enrichment → Filtering → Projection → Batching → Send
  1. Queue: every event produced by Otoroshi is published to all active exporters. Each exporter holds an in-memory bounded queue (bufferSize, default 5000). When the queue is full, the oldest events are dropped to keep the system responsive.
  2. JSON enrichment: events are converted to JSON and enriched with geolocation, user-agent parsing, and instance metadata. The jsonWorkers parameter controls the parallelism of this step.
  3. Filtering: include/exclude rules are applied (see Matching and projections). Events that do not match are discarded.
  4. Projection: if a projection is defined, only the selected fields are kept in the JSON payload.
  5. Batching: events are grouped into batches by count (groupSize, default 100) or by time (groupDuration, default 30 seconds), whichever comes first. This reduces the number of network calls to the destination.
  6. Send: batches are sent to the configured destination. The sendWorkers parameter controls how many batches can be sent concurrently. If a send fails, the events are re-queued for retry.

Tuning tips​

  • High-traffic environments: increase bufferSize to avoid dropping events, increase jsonWorkers and sendWorkers for more parallelism, and increase groupSize for fewer but larger network calls.
  • Low-latency alerting: decrease groupDuration (e.g. 5000 ms) so batches are sent more frequently.
  • Resource-constrained setups: keep worker counts low and use filtering aggressively to reduce the volume of events processed.

Lifecycle​

Exporters are hot-configurable. When you create or update an exporter through the admin API or the UI, Otoroshi gracefully drains the old pipeline (letting in-flight batches complete) and starts a new one with the updated settings. No restart is required.

Example: only keep Otoroshi alerts:

Projection is a list of fields to export. If empty, all fields are exported. If specified, only the listed fields will be included in exported events.

      "value": "v1"
    }
  ],
  "scheme": "HTTPS"
}

$header

Description

{
  "<target>": {
    "$header": {
      "path": "<path>",
      "name": "<name>",
      "default": "<default_value>"
    }
  }
}

Put specified header value in <target>. <path> indicate an array of key/value headers, <name> indicate the name of the header.

Expression

{
  "hostValue": {
    "$header": {
      "path": "headers",
      "name": "Host"
    }
  }
}

Event

{
  "headers": [
    {
      "key": "Host",
      "value": "otoroshi.oto.tools:9999"
    },
    {
      "key": "Accept",
      "value": "application/json"
    }
  ]
}

Result

{
  "hostValue": "otoroshi.oto.tools:9999"
}

$includeAllKeysMatching

Description

{
  "$spread": true,
  "<subname>": {
    "$includeAllKeysMatching": [
      "<expression>"
    ]
  }
}

Filter an object entries based on StartsWith, Wildcard or Regex expression. A logical OR will be applied between filters.

Expression

{
  "$spread": true,
  "_": {
    "$includeAllKeysMatching": [
      "Wildcard(f*)",
      "StartsWith(fi)",
      "bar",
      "Regex(.*lo)"
    ]
  }
}

Event

{
  "foo": 1,
  "foobar": 2,
  "bar": 3,
  "baz": 4,
  "hello": "world",
  "fifou": "test"
}

Result

{
  "foo": 1,
  "foobar": 2,
  "bar": 3,
  "hello": "world",
  "fifou": "test"
}

$excludeAllKeysMatching

Description

{
  "$spread": true,
  "<subname>": {
    "$excludeAllKeysMatching": [
      "<expression>"
    ]
  }
}

Filter an object entries, excluding them based on StartsWith, Wildcard or Regex expression. A logical OR will be applied between filters.

Expression

{
  "$spread": true,
  "_": {
    "$excludeAllKeysMatching": [
      "Wildcard(fo*)",
      "StartsWith(fi)",
      "bar",
      "Regex(.*lo)"
    ]
  }
}

Event

{
  "foo": 1,
  "foobar": 2,
  "bar": 3,
  "baz": 4,
  "hello": "world",
  "fifou": "test"
}

Result

{
  "baz": 4
}

$jq

Description

{
  "<target>": {
    "$jq": "<jqExpression>"
  }
}

Fill target field with provided JQ selector

Expression

{
  "headerKeys": {
    "$jq": "[.headers[].key]"
  }
}

Event

{
  "headers": [
    {
      "key": "k1",
      "value": "v1"
    },
    {
      "key": "k2",
      "value": "v2"
    }
  ]
}

Result

{
  "headerKeys": [
    "k1",
    "k2"
  ]
}

$jqIf

Description

{
  "<target>": {
    "$jqIf": {
      "filter": "<jqExpression>",
      "predicate": {
        "path": "<path>",
        "value": "<value>"
      }
    }
  }
}

Fill target field with provided JQ selector if and only if value located at <path> is equal to <value>

Expression

{
  "headerKeys": {
    "$jqIf": {
      "filter": "[.headers[].key]",
      "predicate": {
        "path": "target.scheme",
        "value": "https"
      }
    }
  }
}

Event

{
  "headers": [
    {
      "key": "k1",
      "value": "v1"
    },
    {
      "key": "k2",
      "value": "v2"
    }
  ],
  "target": {
    "scheme": "https"
  }
}

Result

{
  "headerKeys": [
    "k1",
    "k2"
  ]
}

Sends events in batch to an HTTP endpoint using POST with a JSON array body.

PropertyTypeDescription
